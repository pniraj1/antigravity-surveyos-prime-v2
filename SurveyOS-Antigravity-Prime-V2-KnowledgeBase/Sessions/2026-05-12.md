> [!info] Session Summary
> Fixed a critical navigation bug where clicking "Dashboard" from inside a Claim left the user permanently stuck in the Claim view. Root cause was a race condition in the bi-directional URL sync hook (`useRouteSync`). Both the hook and the sidebar were patched. Deployed to production.

## What We Did

1. **Diagnosed the race condition** — Traced the bug through `sidebar.tsx`, `claim-store.ts`, `claimSlice.ts`, and `useRouteSync.ts` to identify the exact failure point.
2. **Fixed `useRouteSync.ts` (primary fix)** — Added a stale-URL guard in Effect 1 (URL→Store). When `currentClaimId` is already `null` in the store (closeClaim was called) but the URL still has `?claim=XXX` (Effect 2 hasn't fired yet), the effect now bails out immediately instead of re-loading the claim.
3. **Fixed `sidebar.tsx` (belt-and-suspenders)** — Batched `closeClaim()` and `setActiveTab('dashboard')` into a single synchronous block with an early `return`. This ensures React processes both state mutations in one render cycle, shrinking the race window to near-zero.
4. **Updated `Modules/State_Management.md`** — Documented the `claim-store` action table, `useRouteSync` two-effect architecture, and the race condition fix with full explanation.
5. **Built and deployed** — Clean production build (TypeScript exit 0) + deployed to Firebase Hosting.

## Root Cause (Technical Detail)

> [!warning] Race Condition — `useRouteSync`
> Two `useEffect` hooks in `useRouteSync.ts` compete:
> - **Effect 1 (URL→Store):** Watches `searchParams`. If URL has `?claim=XXX`, calls `loadClaim()`.
> - **Effect 2 (Store→URL):** Watches Zustand state. When `currentClaimId = null`, removes `?claim` from URL.
>
> When user clicked Dashboard: `closeClaim()` cleared the store, but before Effect 2 could push the clean URL, Effect 1 fired on the stale `?claim=XXX` URL and restored the claim. The stale-URL guard in Effect 1 breaks this cycle.

## Key Decisions

> [!tip] Key Takeaways
> - `claimSlice.ts` is a **Zustand slice**, NOT Redux. The app is 100% Zustand.
> - Always call `closeClaim()` (never `set({currentClaim: null})`) when exiting a claim — it handles both the claim store AND the ui-store's `currentClaimId` in a single atomic update.
> - `useRouteSync` is the authoritative URL sync layer. State changes flow: Zustand → Effect 2 → URL → Effect 1 → Zustand (but the guard short-circuits this last step when clearing).
