## Summary

Redesigned the Insured Report generation pipeline to eliminate blank narrative sections and enforce data quality before generation. Introduced a 10-category deduction taxonomy as a single source of truth, a deterministic pre-generation gate that blocks report creation until surveyors add remarks for ambiguous rows, a deterministic pre-classifier for obvious categories (safe, salvage, negotiated), and an honest fallback mechanism that produces professional amount-referenced explanations for any row the AI still cannot explain.

---

## Changes

### New File
- `src/lib/constants/deduction-categories.ts` — **Single source of truth** for all 10 deduction categories. Exports `DeductionCategory` type, `DEDUCTION_CATEGORIES` array, `CATEGORY_BADGE_LABELS` and `CATEGORY_BADGE_COLOURS` maps. All AI prompts, types, and badge UI import from here.

  | Category | Meaning |
  |---|---|
  | `depreciation` | Standard wear based on vehicle age |
  | `consumable` | Excluded consumables (coolant, oil, etc.) |
  | `negotiated` | Labour/paint rate negotiated down |
  | `not-covered` | Item excluded under policy |
  | `previous-damage` | Pre-existing damage unrelated to accident |
  | `safe` | Part assessed as safe, no replacement |
  | `salvage` | Disposal/scrap parts (value deducted) |
  | `partial-repair` | Repair assessed instead of replacement |
  | `wear-and-tear` | Age/use degradation, not accident damage |
  | `overpricing` | OEM price reduced to OES/market rate |

### Modified Files
- `src/types/insured-report.ts` — `DeductionCategory` now imported locally from constants and re-exported so all downstream imports remain unchanged.

- `src/lib/ai/insured-report.ts`
  - `getBlockingRows()` — synchronous gate function. Scans all assessment rows and returns a list of items requiring remarks before report generation. Blocks on: disallowed rows with no remarks, and assessed < billed (non-depreciation) with no remarks. Depreciation is exempt because the policy rule is deterministic.
  - `buildPreClassifiedExplanations()` — deterministic classifier that handles `safe`, `salvage`, and `negotiated` rows in TypeScript, skipping the AI entirely for these obvious cases. Reduces token usage and prevents hallucination.
  - `buildHonestFallback()` — when AI marks a row as `isFlagged`, generates a professional, amount-referenced fallback string (e.g. "Assessed at ₹X against billed ₹Y — surveyor review pending"). Eliminates blank PDF sections.

- `src/lib/ai/prompts.ts`
  - Added 3 new categories (`partial-repair`, `wear-and-tear`, `overpricing`) to AI instruction enum.
  - Added **inference-first rule**: AI must try numeric signals before flagging — `assessed ≈ billed → safe`, `labour/paint billed > assessed → negotiated`, `zeroDep=false AND assessed < billed → depreciation`, `isDisposal=true → salvage`. Only flag if inference is inconclusive.
  - Updated `CATEGORY_LABELS` map to include all 10 categories.

- `src/components/tabs/InsuredReportTab.tsx`
  - Imported `getBlockingRows` and `CATEGORY_BADGE_*` from new constants.
  - Added `zeroDep` derivation from claim policy type.
  - **Gate panel** (red block, appears pre-generation): lists every blocking row with its `particulars`, `reason`, and actual ₹ billed → assessed amounts.
  - **Generate button** disabled + label changes to *"Add remarks for N items first"* while gate is active.
  - Flagged-items warning updated: now references honest fallback rather than generic note.

## Design Decisions

> [!info] Why gate in TypeScript, not AI?
> The gate logic is entirely deterministic — it checks for the presence of `remarks` strings against known structural conditions (`action=disallow`, `assessed < billed`, `zeroDep`). Using AI for this would add latency and risk false negatives. Determinism first, AI only for genuine semantic analysis.

> [!info] Why exempt depreciation from the gate?
> Depreciation is a policy-automatic deduction based on vehicle age. The surveyor has no additional context to add; the policy table determines the deduction. Adding a remarks requirement would be busywork with no quality benefit.

> [!info] Honest fallback vs generic note
> The previous behaviour was `isFlagged → blank explanation` in the PDF. The new fallback generates a sentence like "The assessed amount of ₹X differs from the garage estimate of ₹Y — this deduction is under review by the surveyor." This is always true, always professional, and never leaves the insured with a blank section.
